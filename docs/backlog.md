## Journal re-sync — one-time backfill of pre-D2 entries — shipped
Brief: [journal-resync-patch.md](journal-resync-patch.md). D2 made journal entries sync, but pre-existing entries were stranded: the in-memory dirty registry starts empty on load, and pre-D2 the client pushed entries and marked them clean (`stampMap` includes `journalEntries`) even though the old server silently dropped them — so every existing entry sits in localStorage flagged "synced" while the server never stored it.
- **`persistence.markAllJournalEntriesDirty()`** — re-flag every `cache.journalEntries[].id` dirty (incl. soft-deleted, so tombstones travel); touches neither the cache nor `updatedAt`.
- **`sync.ts` trigger** — after a `/sync` response, if `'journalEntries' in resp.pull` (the new server always sends the key; the old one never did) AND `writer-studio-journal-resync-v1` is unset → backfill once, set the flag, and fire an immediate follow-up sync to push the backlog promptly. Idempotent per device; LWW + stable ids make re-pushing a row the server already has a no-op.
- Harness double upgraded to mirror the post-D2 pull shape (always carries a `journalEntries` key) + capture pushes via `/api/_state`. Verified in-harness (4 checks: flag set, the stranded entry re-pushes, flag persists, a second run is prevented). `tsc` + `build:web` + selftest green. The pre-existing-entry case the throwaway round-trip can't reach by construction.
